## vault-scaffold

Seed a fresh vault with the standard OMS folder structure and governance stubs.

### When to use

Use `vault-scaffold` once when initialising a new vault against the OMS engine.
Re-running on an existing vault is safe ‚Äî all writes are additive; existing files are
never overwritten.

### Inputs

- `vaultRoot` ‚Äî absolute path to the target vault directory
- `taxonomySource` ‚Äî path to the engine DEFAULT taxonomy (default: `core/ontology/taxonomy.yaml`)
- `overwrite` ‚Äî boolean (default: `false`); when `false`, skip any path that already exists

### Recipe

1. Read `core/ontology/taxonomy.yaml` to load the engine DEFAULT folder‚Üíconcept map.
2. For each folder declared in `taxonomy.yaml`, create the directory under `vaultRoot` if it
   does not already exist (e.g., `references/`, `notes/`, `inbox/`).
3. Create `vaultRoot/decisions/` ‚Äî a zone for human-authored structural decision records,
   generalising the Ataraxia "95. Decisions" pattern; consumed by the `vault-decision-record`
   skill at runtime.
4. Write `vaultRoot/.oms/taxonomy.yaml` as the per-vault override path.
   Initial content is a verbatim copy of the engine DEFAULT; the vault owner customises it
   here without touching the engine source.
5. Write `vaultRoot/.oms/governance/architecture.md` stub.
   Minimum required content:

   ```markdown
   # Vault Architecture

   > Auto-generated by vault-scaffold v0.1.0. Edit freely.

   ## Taxonomy source

   Override: `.oms/taxonomy.yaml`
   Engine default: `core/ontology/taxonomy.yaml`

   ## Zones

   <!-- List structural zones and their intent here. -->
   ```

6. Do not perform any vault mutations at build time or skill-load time.
   This recipe describes runtime behaviour executed when the skill is explicitly invoked.

### Output

Returns a scaffold report:

- `created` ‚Äî list of paths written
- `skipped` ‚Äî list of paths that already existed (when `overwrite: false`)
- `taxonomyZones` ‚Äî folder names resolved from the engine DEFAULT

### Constraints

- Never writes outside `vaultRoot`.
- Never modifies `core/ontology/taxonomy.yaml` or `core/ontology/concepts/*.yaml`
  (Layer 1 CONTRACT files ‚Äî read-only from this skill).
- The `.oms/taxonomy.yaml` written here is the OVERRIDE path; the engine DEFAULT
  remains authoritative until the vault explicitly diverges.
- Folder creation is `mkdir -p` semantics; missing intermediate directories are created.
